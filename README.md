## 1. 프로젝트 소개


###  🔎 Klüme는 어떤 서비스인가?

Klüme은 다음과 같은 실제 문제 상황을 해결하기 위해 기획되었습니다.

#### ❌ 기존 회의실 예약 문제
- 선착순 구두 예약 → 줄서기, 혼잡
- 매니저 출근 전까지 자리확보 대기
- 가위바위보로 자리 배분 → 운에 따른 불공정
- 예약 현황 확인이 어려움
- 공지가 디스코드 등 여러 곳으로 흩어져있음

#### ✅ Klüme가 제공하는 해결책
- ⏱ 정확한 예약 오픈 시간 + 자동 활성화 버튼(티켓팅 방식)
- ⚡ 여러 명 클릭 시 서버 단에서 단 1명만 예약 성공 처리
- 📅 일/주/월 달력 기반 예약 현황 제공
- 🏢 조직/그룹 단위로 예약 환경 관리
- 💬 실시간 관리자–멤버 채팅
- ⚠️ 패널티 기반 질서 유지(노쇼 방지, 인증 사진 업로드)

## 9. 개인 회고

### 강지륜
**📚 프로젝트에서 맡은 역할**
- 프로젝트 기본 세팅
- 조직 관련 전반적인 기능 구현
   - 조직 및 그룹 생성 / 수정 / 삭제 기능 구현
   - 조직 초대 코드 발급 및 가입 기능 구현
   - 조직 멤버 수정 / 제거 등 관리 기능 구현
- 회의실 예약 기능 구현

**🏆 잘한 점**
- 프로젝트 초기 세팅을 담당하며 팀원들이 동일한 환경에서 작업할 수 있도록 기반을 마련했다. 
- 서비스 클래스에서는 메소드 책임을 최소화해 추상화 수준을 통일하려고 노력했고 메소드 이름만 보고도 기능을 파악할 수 있도록 코드를 작성했다. 
- 코드 리뷰를 통해 팀원들의 피드백을 반영하여 오류를 개선하고 코드 품질을 높일 수 있었고 다른 팀원들의 코드를 리뷰하면서 전반적인 코드에 대한 이해도도 높였다.

**🤨 아쉬운 점**
- 동시성 제어를 프로젝트 후반부에 도입하다보니 충분히 학습하지 못한 상태에서 적용하여 코드가 제대로 구현되었는지에 대한 확신이 부족했다.
- 백엔드 기능 구현에 집중하느라 프론트 개발에 충분한 시간을 투자하지 못했다.
- 기능 구현을 우선하다 보니 테스트 코드를 병행해 작성하지 못했다.

### 김진호
**📚 프로젝트에서 맡은 역할**
- 프로젝트 전반 기획 참여
- ERD 및 전체 모델링 설계
- 웹소켓 기반 채팅 기능 구조 및 전반 관리(STOMP, SockJS 포함)
- 프론트·백엔드 간 API 및 데이터 구조 조율
- 채팅 기능 UI/UX 개발 및 수정
- 형상관리(GitHub) 브랜치 전략·PR 규칙 관리 및 팀원 코드 리뷰 참여
  
**🏆 잘한 점**

이전에 경험해보지 않았던 웹소켓·STOMP·SockJS와 같은 실시간 통신 기술과 몽고DB 같은 NoSQL 기술들을 직접 학습하고, 왜 해당 기술들을 사용해야 하는지 구조적·기능적 관점에서 스스로 고민하고 문서화시켜 팀원들과도 공유하고 적용했다는 점이 큰 성취였다. 특히 메시지가 브로커를 통해 어떻게 라우팅되는지, 구독/발행 구조가 어떻게 동작하는지, 폴백이 필요한 이유가 무엇인지 등을 이해하며 실시간 기능을 안정적으로 구현했고, 이러한 학습 과정을 팀원들과도 공유하며 프로젝트 품질을 높일 수 있었다. 또, MVP를 먼저 구현한 뒤 채팅 플로팅 띄우기, 채팅 담당하기, ui/ux 개선등 점진적으로 기능을 확장하는 방식으로 개발해 리스크를 줄이고 빠른 검증을 할 수 있었던 점도 스스로 좋은 선택이었다고 느낀다. 또한 GitHub 브랜치 전략, 커밋 규칙, PR 리뷰 프로세스 등 팀이 합의한 형상관리 원칙과 코딩 컨벤션을 일관되게 지키며 협업 효율을 높이고, 팀 전체 코드가 동일한 스타일과 흐름을 유지하도록 신경 쓴 점도 좋았다.


**🤨 아쉬운 점**

설계 단계에서 최대한 자세하고 명확히 하려고 노력했지만 구현 단계에서 많이 바뀌고 재설계 단계를 거쳐서 시간이 다소 지연된것 같다.
SSE 기반의 실시간 알림 기능을 완성하지 못한 점이 아쉽다.
웹소켓과 실시간 통신에 대한 깊은 이해보다는 시간에 쫓겨 구현에만 급급한 모습이 있었던 것 같다.

**🎯 배운 점**

- 형상관리 규칙과 코드리뷰를 팀 전체가 일관되게 지키는 것의 중요성을 체감했다.
-  HTTP API와는 전혀 다른 방식으로 동작하는 웹소켓 기반 실시간 기능을 STOMP 프레임 구조, 메시지 라우팅, 세션·구독관리등을 직접 설계·구현하면서 이해하게 되었다.
- MongoDB를 사용하면서 문서 기반 저장의 장점, 쿼리 방식, 데이터 모델링의 관점을 새롭게 익힐 수 있었다.
- 새롭게 배우게 되는 기술에 대한 막연한 두려움이 없어지고 "하면 되겠지"라는 자신감을 얻을 수 있었다.

**💡 다음 프로젝트에 적용하고 싶은 점**

초기에 MVP를 빠르게 완성하고 그 위에 기능을 자연스럽게 확장해가는 방식을 더 체계적으로 적용하고 싶다. 실시간 통신 기능을 다시 구현한다면 웹소켓뿐 아니라 SSE 기반 알림 시스템, Redis Pub/Sub, 메시지 큐, 읽음 처리·오프라인 메시지 관리 등 실서비스에 더 가까운 구조를 적용하고 싶다. 또한 프론트와 백엔드가 더 빠른 템포로 협업할 수 있도록 API와 이벤트 문서화를 정착시키고, 초반 단계에서 end-to-end 흐름을 빠르게 검증하는 방향으로 진행하고 싶다.

### 배태용
**📚 프로젝트에서 맡은 역할**

- 회원가입(로컬, Google OAuth) 기능 개발

- Spring Security 기반 로그인 인증·인가 처리

- 공통 컴포넌트(header, sidebar, button, input) 제작

- Front-End 요구사항에 맞춘 Back-End DTO 추가 및 API 구조 정리

이번 프로젝트는 회원 도메인을 중심으로 전체 인증 흐름을 책임졌고, 클라이언트 개발과의 연결도 꽤 밀접하게 진행했다.

**🏆 잘한 점**
- Spring Security를 활용한 인증/인가 구축

이번 프로젝트에서 내가 가장 많은 시간을 투자한 부분이었다. Spring Security는 여전히 난이도가 높은 영역이라 처음 구조를 잡는 과정부터 시행착오가 많았다.
수업 때는 따라가기 바빴지만, 실제로 프로젝트에서 직접 구현하면서 구조를 이해할 수 있었고, 인증 흐름을 내 손으로 제대로 구성해볼 수 있었다는 점이 가장 큰 성취였다.
특히 Local 회원가입뿐 아니라 Google OAuth까지 확장하면서 단순히 API 연동이 아니라,
- Spring Security 필터 구조
- OAuth2 클라이언트 설정
- 사용자 정보 매핑
- 회원 테이블과 OAuth 사용자 정보 동기화

등 여러 포인트를 경험할 수 있어서 실제 서비스 구성에 한 발 더 다가간 느낌을 받았다.
처음엔 어려웠지만 결국 완성해서 팀 전체가 안정적으로 사용할 수 있었다는 점이 꽤 뿌듯했다. 😎

**🤨 아쉬운 점**

Google OAuth에 이어 카카오/네이버 같은 국내 OAuth를 적용하지 못한 점은 개인적으로 많이 아쉬웠다.
OAuth 구조 자체는 비슷하지만, Provider별로 인증 방식이나 반환 데이터가 달라 실제로 붙여보는 경험을 해보고 싶었다.
DevOps 프로젝트였지만, 실제 개발에서는 DevOps 역할을 많이 맡지 않아 CI/CD, 인프라 구성 등에 깊게 관여하지 못했다. 
수업을 들은 것과 실전 프로젝트에서 직접 다뤄보는 것은 차이가 크기 때문에, DevOps 실전 경험을 쌓지 못한 점이 지금 돌아보면 아쉽다.

**🎯 배운 점**

- Redis 활용 능력 향상

이번에도 회원가입 인증 이메일 발송 후 인증 제한 시간을 관리하기 위해 Redis를 사용했다.
지난 프로젝트에서는 다른 팀원이 개발해 설치된 Redis를 그냥 사용하는 수준이었다면, 
이번에는 직접 회원가입 기능을 맡으면서 TTL 설정, key/value 구조 설계, 인증 번호 관리 로직 등을 더 깊이 있게 이해하고 사용할 수 있었다.

**💡 다음 프로젝트에 적용하고 싶은 점**

단위 테스트 코드 작성 습관 개선
기능을 완성한 후 수동 테스트에만 의존하다 보니 리팩토링 시 불안함이 있었고, 이번 경험을 통해 테스트 코드의 필요성을 다시 체감했다.
다음 프로젝트에서는 인증/인가, 레포지토리, 서비스 레이어 중심으로 단위 테스트를 적극적으로 작성하고 싶다.

디자인 패턴 활용 능력 향상
이번에 인증 관련 로직을 구현하면서 중복 코드가 생기거나 확장성 있는 구조를 만들지 못한 부분이 조금 아쉬웠었다. 
이 다음 최종 프로젝트 때엔 전략 패턴, 템플릿 메서드 패턴 등 상황별로 적용해볼 수 있는 패턴들을 더 적극적으로 고민해보고 싶다.
